Activity Lifecycle & Callbacks

Core

The Android activity lifecycle consists of a series of callback methods that get called as an activity transitions between different states: Created, Started, Resumed, Paused, Stopped, and Destroyed.

Example:
override fun onCreate(savedInstanceState: Bundle?) { super.onCreate(savedInstanceState) setContentView(R.layout.activity_main) // Initialize components } override fun onStart() { super.onStart() // Activity becoming visible } override fun onResume() { super.onResume() // Activity in foreground }

onPause() vs onStop() vs onDestroy()

Core

onPause(): Called when activity is partially obscured (still visible). Save transient data here.

onStop(): Activity no longer visible. Release resources that aren't needed while app is hidden.

onDestroy(): Final call before activity is destroyed. Clean up resources to prevent leaks.

View vs ViewGroup

UI & Layout

View: Basic building block for UI components (Button, TextView, etc.). Responsible for drawing and event handling.

ViewGroup: Subclass of View that contains other views (layouts). Responsible for arranging child views.

XML Example:
<LinearLayout> <!-- ViewGroup --> <TextView /> <!-- View --> <Button /> <!-- View --> </LinearLayout>

ConstraintLayout Benefits

UI & Layout

ConstraintLayout improves performance by:

  • Reducing view hierarchy depth (flat layout)
  • Eliminating nested layouts
  • Using constraint-based positioning
  • Providing better control over view placement
XML Example:
<androidx.constraintlayout.widget.ConstraintLayout> <Button app:layout_constraintStart_toStartOf="parent" app:layout_constraintTop_toTopOf="parent"/> <TextView app:layout_constraintStart_toEndOf="@id/button" app:layout_constraintTop_toTopOf="parent"/> </androidx.constraintlayout.widget.ConstraintLayout>

Jetpack Compose Principles

Jetpack Compose

Core principles of Jetpack Compose:

  • Declarative: Describe UI based on current state
  • Composable functions: Build UI with annotated functions
  • Recomposition: Efficient UI updates when state changes
  • Interoperability: Works with existing View system
Example:
@Composable fun Greeting(name: String) { Text(text = "Hello, $name!") }

MVVM Benefits

Architecture

Benefits of MVVM over MVC/MVP:

  • Separation of concerns (UI, business logic, data)
  • Lifecycle awareness with ViewModel
  • Data binding reduces boilerplate code
  • Easier unit testing of ViewModels
  • Better handling of configuration changes
MVVM Structure:

View (Activity/Fragment) ↔ ViewModel ↔ Model (Repository)

Room Components

Data Persistence

Room Persistence Library components:

  • @Entity: Represents a table in the database
  • @Dao: Data Access Object - contains methods to access the database
  • @Database: Database holder - serves as the main access point
Example:
@Entity data class User( @PrimaryKey val id: Int, val name: String ) @Dao interface UserDao { @Query("SELECT * FROM user") fun getAll(): List<User> } @Database(entities = [User::class], version = 1) abstract class AppDatabase : RoomDatabase() { abstract fun userDao(): UserDao }

Retrofit Annotations

Networking

Common Retrofit annotations:

  • @GET, @POST: Define HTTP request type
  • @Path: Replace placeholders in URL
  • @Query: Add query parameters
  • @Body: Send request body (for POST/PUT)
  • @Header: Add headers to request
Example:
interface ApiService { @GET("users/{id}") fun getUser(@Path("id") userId: String): Call<User> @GET("users") fun getUsers(@Query("page") page: Int): Call<List<User>> @POST("users") fun createUser(@Body user: User): Call<User> }

Kotlin Coroutines Basics

Concurrency

Key coroutine concepts:

  • launch: Start a new coroutine without returning result
  • async: Start a coroutine that returns a result (Deferred)
  • withContext: Switch coroutine context (e.g., to IO thread)
Example:
viewModelScope.launch { // Main thread val data = withContext(Dispatchers.IO) { // Background thread - fetch data repository.fetchData() } // Back on Main thread - update UI _uiState.value = data }

Kotlin Null Safety

Kotlin

Kotlin's null safety features:

  • ?: Nullable type (String?)
  • !!: Non-null assertion (throws NPE)
  • ?.: Safe call operator (returns null if receiver is null)
  • ?:: Elvis operator (provide default value)
  • lateinit: For non-null properties initialized later
  • by lazy: Deferred initialization
Example:
val length = text?.length // returns null if text is null val safeLength = text?.length ?: 0 // returns 0 if null lateinit var user: User // will be initialized later val db: Database by lazy { // Initialized on first access createDatabase() }

RecyclerView Optimization

Performance

Optimizing RecyclerView performance:

  • Always use ViewHolder pattern
  • Use DiffUtil for efficient updates
  • Implement ListAdapter for automatic diffing
  • Set setHasFixedSize(true) when possible
  • Enable view caching and prefetching
  • Avoid nested scrolling layouts
DiffUtil Example:
class UserDiffCallback : DiffUtil.ItemCallback<User>() { override fun areItemsTheSame(old: User, new: User) = old.id == new.id override fun areContentsTheSame(old: User, new: User) = old == new } val adapter = object : ListAdapter<User, ViewHolder>(UserDiffCallback()) { // Adapter implementation }
